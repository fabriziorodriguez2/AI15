1 · MVP operativo

Una solución concreta que puede probarse

AI15 es una aplicación mobile para organizar fiestas de quince. Centraliza presupuesto, invitados, tareas, decisiones, proveedores e inspiración, y convierte esos datos en un plan personalizado mediante inteligencia artificial.

14rutas de producto y API
2features principales de IA
1usuario demo reproducible

¿Por qué es un MVP operativo?

  • Se ejecuta localmente con instrucciones documentadas.
  • Permite recorrer la experiencia sin login.
  • Incluye datos de ejemplo listos para probar.
  • Registra gastos y calcula saldo real.
  • Administra invitados, tareas y decisiones.
  • Genera un plan real mediante Gemini.
  • Genera una distribución presupuestal con IA.
  • Busca proveedores reales mediante Places si se configura la key.

Necesidad concreta

La información de una fiesta suele quedar dispersa entre chats, capturas, notas y planillas, generando incertidumbre y estrés familiar.

Valor aportado

AI15 transforma información dispersa en una visión única del presupuesto, el avance y los próximos pasos.

Nivel alcanzado: MVP funcional y testeable; si se considera su antecedente del Obligatorio 1, corresponde describirlo como una beta evolucionada.

5 · IA real y justificada

La IA resuelve una combinación difícil de variables

La feature de IA no decora la experiencia: interpreta fecha, presupuesto, gastos, decisiones y estilos para producir una guía que una plantilla genérica no podría adaptar con la misma flexibilidad.

Plan integral

Genera síntesis, asignación presupuestal, próximas tareas, propuesta estética, recomendaciones y advertencias.

Distribución inteligente

Propone porcentajes por categoría y explica prioridades según la fiesta cargada.

Servicio y modelo

ProveedorGoogle Gemini mediante proxy estudiantil de Vertex.
Modelogemini-2.5-flash.
RolSíntesis contextual y recomendación; no reemplaza cálculos ni decisiones familiares.
InteracciónBotones específicos, modal de consentimiento y resultados estructurados; no chat abierto.

Justificación

  • Relaciona múltiples datos del mismo evento.
  • Prioriza según días restantes.
  • Mantiene coherencia estética.
  • Respeta decisiones confirmadas.
  • Adapta el presupuesto al contexto.
  • Explicita incertidumbre mediante warnings.
Límite de responsabilidad: las respuestas son orientativas. La familia debe validar costos, contratos, disponibilidad y decisiones finales.

Zod + UI

1. Contexto curado

El frontend selecciona solo los campos necesarios, sin enviar imágenes ni contactos.

2. Validación de entrada

La ruta valida estructura, cantidades y longitudes antes de llamar al modelo.

3. Generación en servidor

La key permanece en variables de entorno y el navegador nunca contacta directamente al modelo.

4. Respuesta estructurada

Gemini recibe un JSON Schema. La respuesta se parsea incluso si viene envuelta en fences Markdown.

5. Segunda validación

Zod verifica campos, fechas, prioridades y límites. Los porcentajes se normalizan y comprueban en TypeScript.

6. Memoria de sesión

El resultado se muestra y se conserva durante la sesión. Si cambian datos importantes queda marcado como desactualizado.

Decisión técnica surgida de una prueba real

El proxy truncaba respuestas largas. El plan se dividió en dos generaciones breves ejecutadas en paralelo: una logística y otra creativa. Luego se combinan y validan como un único plan. Así se evita el error 502 sin inventar contenido faltante.

Decisiones de diseño y curaduría

DecisiónMotivo
Sin chat abiertoReducir ambigüedad y conectar la IA con acciones concretas del producto.
JSON Schema + ZodEvitar respuestas imposibles de mostrar o guardar.
Respuesta breveEvitar truncamiento, mejorar lectura mobile y reducir costo.
Decisiones confirmadas como restriccionesNo contradecir elecciones ya tomadas por la familia.
No inventar proveedoresSeparar recomendaciones generales de negocios verificables.
Cálculos en TypeScriptNo confiar al modelo la suma exacta del presupuesto.

Recorrido recomendado

  1. Avanzar por las tres pantallas del onboarding.
  2. Tocar Ver demo.
  3. Revisar dashboard y cronograma.
  4. Agregar un gasto y comprobar el saldo.
  5. Generar una distribución desde Presupuesto.
  6. Generar el plan personalizado desde Plan.
  7. Agregar las tareas sugeridas al cronograma.
  8. Ir a Proveedores → Ejemplos y probar los filtros del catálogo hardcodeado.

11 · API keys y seguridad

Las credenciales nunca llegan al navegador

  • Claves almacenadas en .env.local.
  • .env.local está ignorado por Git.
  • Ninguna key usa prefijo NEXT_PUBLIC_.
  • Cliente Gemini marcado server-only.
  • Solo existen variables privadas para Gemini.
  • Proveedores no requiere credenciales externas.
  • Las claves no se registran en logs.
  • Los errores externos se vuelven mensajes genéricos.
  • Timeouts de red evitan esperas indefinidas.
  • Entradas limitadas y validadas con Zod.
  • Las respuestas de IA se validan antes de usarse.
  • Los datos demo no contienen información sensible.
